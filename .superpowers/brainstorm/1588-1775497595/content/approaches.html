Three Architecture Approaches

How to integrate agentic AI + cinematic visuals into AstroWatch

A

Unified Agent Gateway

Single API layer routes to both models. One /api/agent endpoint acts as the brain. It decides whether to call Qwen (chat) or Claude (analysis/visuals) based on intent classification. The autonomous agent is a Vercel cron that calls the same gateway.


Pros

  • Simple client code — one endpoint for everything
  • Easy to swap models later
  • Centralized cost tracking

Cons

  • Single point of failure
  • Intent classification adds latency + complexity
  • Harder to optimize each path independently
B

Dual-Model Split Architecture (Recommended)

Separate API routes for chat vs. autonomous agent. /api/chat handles conversational interactions via Qwen with tool-calling (scene control, data queries). /api/agent is the autonomous Claude-powered analyst that runs on schedule, publishes threat assessments, and generates scene annotations. They share a common data layer but are independently deployable.


Pros

  • Each path optimized for its model & use case
  • Chat stays fast (no routing overhead)
  • Autonomous agent can evolve independently
  • Clear separation of concerns — great CV talking point
  • If one model is down, the other still works

Cons

  • Two integration paths to maintain
  • Need shared context layer for cross-model awareness
C

Agent SDK Orchestrator

Use Anthropic's Agent SDK as the orchestration layer. Claude is the "brain" that delegates to Qwen for chat tasks and handles analysis itself. The SDK manages tool definitions, conversation state, and multi-step reasoning natively. The autonomous agent uses the same SDK with different tool sets.


Pros

  • Most sophisticated architecture — strongest CV signal
  • Native tool-calling and multi-step reasoning
  • Built-in conversation memory management

Cons

  • All requests go through Claude API first (higher cost)
  • Adds dependency on Anthropic SDK for everything
  • Defeats the purpose of using Ollama for cheap chat
