Tool Definitions for Qwen

Function calling gives the AI control over the app

query_asteroids

Search and filter asteroid data. Params: risk level, size range, date range, sort by. Returns structured asteroid data the AI weaves into natural language.

control_scene

Manipulate the 3D view. Actions: focus(asteroidId), set_filter(risk), set_view(mode), orbit_camera(target, angle), toggle_trajectories. Dispatches to Zustand store.

get_statistics

Compute analytics on demand. Functions: risk_distribution, velocity_histogram, size_comparison, weekly_trend. Returns data the AI can summarize or trigger a chart.

get_agent_insights

Read findings from the autonomous Claude agent. Fetches latest briefings, anomaly alerts, trend observations from Vercel KV. Bridges the two AI systems.

API Route: /api/chat

Protocol: Server-Sent Events (streaming)
Flow: Client sends messages[] → Route calls Ollama Cloud API with tools → Streams text tokens + tool call results back → Client renders incrementally + executes scene commands
System prompt: Includes current asteroid summary, available scene state, latest agent briefing digest. Refreshed per request.
Context window: Last 20 messages. No server-side persistence — conversation lives in client state (Zustand).
Tool execution: Tools run server-side in the API route (data access), results streamed back. Scene control commands sent as special SSE events the client intercepts.

Scene Control Flow

How "zoom in on the closest asteroid" works
1. User types "zoom in on the closest asteroid"
2. /api/chat sends to Qwen with tool definitions
3. Qwen calls query_asteroids({sort: "miss_distance", limit: 1})
4. Route executes, returns asteroid data to Qwen
5. Qwen calls control_scene({action: "focus", id: "2024-BX1"})
6. Route emits SSE: {"type":"scene_command","action":"focus","id":"2024-BX1"}
7. Client intercepts, dispatches to Zustand store → camera transition begins
8. Qwen streams: "Focusing on 2024 BX1 — it's passing at just 0.012 AU..."
